Course Notes — HSL Color Model

▼

Why HSL over HEX? HSL (Hue, Saturation, Lightness) is more intuitive for designers because you can reason about colors: "I want a warm color (hue ~30°) that's vivid (high saturation) and light (high lightness)."

/* In Figma: Use HSL to create systematic palettes */
--primary-50:  hsl(260, 75%, 95%);  /* lightest */
--primary-100: hsl(260, 75%, 90%);
--primary-200: hsl(260, 75%, 80%);
--primary-300: hsl(260, 75%, 70%);
--primary-400: hsl(260, 75%, 60%);  /* base */
--primary-500: hsl(260, 75%, 50%);
--primary-600: hsl(260, 75%, 40%);
--primary-700: hsl(260, 75%, 30%);  /* darkest */

/* Pro tip: Keep hue constant, vary lightness for shade scales */

Figma Technique: In Figma, create color styles using the HSL model. Keep the hue constant and adjust lightness in 10% increments to create a full shade scale. This maps directly to CSS custom properties.

▼

The Type Scale Formula: Pick a base size (16px) and a ratio (1.25 for "Major Third"). Each step is base × ratio^n.

   - Headings: 1.1–1.2

Course Notes — Figma to Code Translation

▼

The Translation Process:

1. Inspect in Figma — Select a layer, look at the right panel for dimensions, colors, fonts, padding, and effects.

2. Map to HTML — Each Figma frame becomes a <div>. Text layers become <p>, <h1-h6>, or <span>. Images become <img>.

3. Map to CSS — Figma's auto-layout → flexbox. Fill → background-color. Drop shadow → box-shadow. Corner radius → border-radius.

/* Figma properties → CSS mapping */
/* Figma: Auto Layout (horizontal, gap: 12, padding: 24) */
display: flex;
gap: 12px;
padding: 24px;

/* Figma: Fill #7c5cfc, Corner Radius 12 */
background: #7c5cfc;
border-radius: 12px;

/* Figma: Drop Shadow X:0, Y:4, Blur:20, #000 8% */
box-shadow: 0 4px 20px rgba(0,0,0,0.08);

Course Notes — Box Model

▼

Critical rule: Always set box-sizing: border-box. This makes padding and border count inside the element's width, not outside.

/* Always add this to your reset */
*,
*::before,
*::after {
  box-sizing: border-box;
}

/* Without border-box:
   width: 200px + padding: 20px + border: 2px = 244px total

   With border-box:
   width: 200px total (padding & border fit inside) */

🎯 Visual — Concentric Border Radii

When nesting rounded elements, the inner radius should be outer radius minus the gap. This creates optically correct, professional-looking nesting.

✓
Correct: 24px → 12px
(24 - 12 gap = 12)
✗
Wrong: 24px → 24px
(same radius = uneven gap)
📝

Course Notes — Concentric Radii Formula

▼ 
/* The golden rule of nested border-radius:
   inner-radius = outer-radius - gap (padding/border)

   If gap >= outer-radius, inner should be 0 */

.outer {
  border-radius: 24px;
  padding: 12px;
}

.inner {
  border-radius: 12px; /* 24 - 12 = 12 ✓ */
}

/* Common use cases:
   Cards inside containers
   Images inside cards
   Buttons inside input groups
   Avatars inside badges */

Course Notes — Elevation Highlight Technique

▼

The Key Insight: In dark UIs, elevated surfaces are lighter, not darker. This mimics how objects closer to a ceiling light reflect more light.

/* Dark UI elevation system:
   Higher elements = lighter surface + stronger shadow */

.surface-1 { background: hsl(240, 20%, 16%); }
.surface-2 { background: hsl(240, 20%, 20%); }
.surface-3 { background: hsl(240, 20%, 24%); }
.surface-4 { background: hsl(240, 20%, 28%); }

/* The top-light highlight using a pseudo-element */
.elevated::before {
  content: '';
  position: absolute;
  inset: 0;
  border-radius: inherit;
  background: linear-gradient(
    180deg,
    rgba(255,255,255,0.08) 0%,
    transparent 50%
  );
  pointer-events: none;
}

Course Notes — Token Naming Best Practices

▼

The Three-Tier Token System:

1. Global tokens — Raw values: --blue-500: #3b82f6

2. Alias tokens — Semantic meaning: --color-primary: var(--blue-500)

3. Component tokens — Scoped usage: --btn-bg: var(--color-primary)

/* Tier 1: Global (raw palette) */
--purple-400: #9b82fc;
--purple-500: #7c5cfc;
--purple-600: #6b4ce0;

/* Tier 2: Alias (semantic) */
--color-primary:       var(--purple-500);
--color-primary-hover: var(--purple-600);
--color-primary-light: var(--purple-400);

/* Tier 3: Component-specific */
--btn-bg:          var(--color-primary);
--btn-bg-hover:    var(--color-primary-hover);
--card-bg:         var(--color-surface);

/* Naming pattern:
   --{category}-{property}-{variant}-{state}
   --color-text-secondary
   --space-padding-lg
   --shadow-elevation-3  */

Figma Tip: In Figma, use "Variables" (or the older "Styles") to define tokens. Name them with slashes for grouping: color/primary/default, color/primary/hover, spacing/sm, spacing/md. This creates the same hierarchy as your CSS custom properties.

Course Notes — Figma Variables → CSS Custom Properties

▼

The Figma → Code Token Pipeline:

In Figma, go to Local Variables (or the older Styles panel). Create collections for Color, Spacing, and Typography. Then map them directly to CSS:

/* Figma Variable Collection: "Color"
   Mode 1: Light    Mode 2: Dark
   ─────────────────────────────────
   primary    #7c5cfc    #9b82fc
   surface    #ffffff    #1a1a26
   text       #1a1a2e    #f0f0f5
*/

/* In CSS, use a class to switch modes: */
:root {
  --color-primary: #7c5cfc;
  --color-surface: #ffffff;
  --color-text:    #1a1a2e;
}

.dark-theme {
  --color-primary: #9b82fc;
  --color-surface: #1a1a26;
  --color-text:    #f0f0f5;
}

/* Components NEVER reference raw colors.
   They only use tokens. When the class
   changes, everything updates. ✨ */

Pro Workflow: Use tools like Tokens Studio (Figma plugin) to export your Figma variables directly as CSS custom properties, JSON, or SCSS variables. This eliminates manual translation entirely.

When → Why → How — Fitts's Law

▼

When to apply: Sizing buttons, touch targets, navigation items, form submit areas — any clickable/tappable element.

Why it matters: Small, distant targets cause frustration and errors. Mobile users with imprecise fingers are especially affected. The law quantifies what we intuit: bigger and closer is easier.

How to implement:

/* Token-based approach: define tap targets once */
:root {
  --tap-target-min: 44px;  /* WCAG minimum */
  --tap-target-comfy: 48px; /* Comfortable */
  --cta-padding: var(--space-3) var(--space-6);
}

/* Apply to all interactive elements */
.btn {
  min-height: var(--tap-target-comfy);
  padding: var(--cta-padding);
}

/* Place primary CTA near thumb zone on mobile */
.mobile-cta {
  position: sticky;
  bottom: var(--space-4);
}

Token savings: Define --tap-target-min once. Every button, link, and icon inherits consistent sizing without repeating values.

When → Why → How — Hick's Law

▼

When to apply: Navigation menus, onboarding flows, pricing pages, form design, settings panels — anywhere users must choose.

Why it matters: Choice overload causes "analysis paralysis." Users abandon interfaces that feel overwhelming. Grouping and progressive disclosure reduce cognitive load without removing functionality.

/* Progressive disclosure: show less, reveal on demand */
.nav-group {
  display: flex;
  flex-direction: column;
  gap: var(--space-1);
}

.nav-group-items {
  display: none; /* Hidden by default */
  padding-left: var(--space-4);
}

.nav-group.expanded .nav-group-items {
  display: flex; /* Reveal on click */
}

Token savings: Reuse --space-* tokens for nested indentation. Use one .expanded class pattern across all collapsible sections.

When → Why → How — Gestalt Principles

▼

When to apply: Every single layout decision. Card grouping, form field arrangement, navigation structure, toolbar organization, dashboard widgets.

Why it matters: The brain automatically groups elements by proximity, similarity, and containment. If your spacing or colors don't match your intended groupings, users will misread your layout.

/* Proximity: Use spacing tokens to signal groups */
.form-group {
  display: flex;
  flex-direction: column;
  gap: var(--space-2);     /* tight = related */
  margin-bottom: var(--space-6); /* wide = new group */
}

/* Common Region: card as a container */
.card {
  background: var(--color-surface);
  border: 1px solid var(--color-border);
  border-radius: var(--radius-lg);
  padding: var(--space-6);
}

/* Similarity: same style = same function */
.btn-group .btn {
  font-size: var(--text-sm);
  padding: var(--space-2) var(--space-4);
}

Token savings: Two spacing tokens create all the grouping you need — a "tight" gap for related items (--space-2) and a "loose" gap for separating groups (--space-6). This is proximity-as-a-system.

When → Why → How — Von Restorff + Serial Position

▼

Von Restorff: Make the most important option visually distinct — different color, size, or elevation.

Serial Position Effect: People best remember the first and last items in a series. Place your most important actions at the start and end of lists, tab bars, and navigation.

📚 Reference — All Laws at a Glance

Click any card to expand the When → Why → How → Token Tip breakdown.

400
ms
Heuristic

Doherty Threshold

"System responses under 400ms keep users engaged."

Productivity soars when both human and computer interact at a pace that ensures neither has to wait on the other.

When to apply

Loading states, transitions, API response handling, animations, skeleton screens

Why it matters

Response times over 400ms break the user's flow state. After 1 second, users start thinking about other things.

How to implement
  • Use CSS transitions with var(--duration-fast): 150ms for micro-interactions
  • Show skeleton screens immediately rather than spinners
  • Use optimistic UI updates (show success before server confirms)
Token savings

Define --duration-fast: 150ms, --duration-normal: 250ms, --duration-slow: 400ms and --ease-out: cubic-bezier(0.16,1,0.3,1). Apply once, consistent everywhere.

⚖️
Principle

Tesler's Law

"Every system has irreducible complexity."

For any system, there's a certain amount of complexity that can't be reduced. The question is: does the user bear it, or does the system?

When to apply

Designing forms, onboarding flows, config screens, or any flow with inherent complexity

Why it matters

If you oversimplify, users can't achieve their goals. If you expose too much complexity, they're overwhelmed. The art is in choosing smart defaults.

How to implement
  • Use smart defaults so most users never need to configure
  • Progressive disclosure: "Advanced options" behind an accordion
  • Absorb complexity in code, not in UI
Token savings

Smart defaults = fewer variant tokens. If 80% of buttons use --btn-md, you don't need --btn-xs through --btn-xxl.

🤝
Principle

Postel's Law

"Be liberal in what you accept, conservative in what you send."

Accept varied user input (formats, casing, whitespace) but output clean, consistent results.

When to apply

Form inputs, search, filters, date fields, phone numbers, addresses — any user input

Why it matters

Users shouldn't have to think about formatting. "2024-01-15", "Jan 15, 2024", and "1/15/24" should all work.

How to implement
  • Auto-format inputs on blur (phone, credit card, dates)
  • Show clear placeholder patterns: placeholder="(555) 123-4567"
  • Trim whitespace, normalize casing, strip special characters
Token savings

One input component with var(--input-padding), var(--input-radius), var(--input-border) handles all formats. No variant styles needed.

Cognitive Bias

Peak-End Rule

"People judge experiences by the peak moment and the ending."

A delightful confirmation screen matters more than every intermediate step being perfect. Invest in peaks and endings.

When to apply

Checkout flows, onboarding completion, form submissions, sign-up success, upgrade moments

How to implement
  • Design a "wow" success state with animation and confetti
  • Add a micro-animation on the final CTA click
  • Make error recovery smooth — that's a "peak" too
Token savings

One --duration-celebration: 600ms token + a reusable .success-state class. Apply to every completion moment without new styles.

Cognitive Bias

Zeigarnik Effect

"Incomplete tasks stick in memory more than completed ones."

Progress bars, step indicators, and checklists motivate completion by creating a sense of unfinished business.

When to apply

Multi-step forms, onboarding checklists, profile completion, course progress, gamification

How to implement
  • Show progress bars with percentage or "Step 2 of 4"
  • Use profile completion prompts: "You're 80% done!"
  • Display checklists with visual check marks
Token savings

One progress bar component: var(--color-primary) fill, var(--radius-full) shape, var(--space-1) height. Reuse everywhere.

Cognitive Bias

Goal Gradient Effect

"People accelerate behavior as they approach a goal."

The closer users are to completing a task, the more motivated they become. Show them how close they are.

When to apply

Checkout flows, loyalty programs, onboarding wizards, upload progress, profile setup

How to implement
  • Start progress at 10-20% (the "endowed progress" trick)
  • Use stepper components showing completed vs. remaining
  • Animate transitions between steps for momentum
Token savings

Stepper tokens: --step-size, --step-active: var(--color-primary), --step-complete: var(--color-success), --step-pending: var(--color-border).

✂️
Principle

Occam's Razor

"The simplest solution is usually the best."

Among competing designs, prefer the one with fewest assumptions and least complexity. Simplify without removing function.

When to apply

Feature design, component architecture, layout decisions, token system design itself

How to implement
  • Audit every element: "Does this earn its place?"
  • Reduce color count — 2–3 primary colors maximum
  • Use fewer font weights (400, 600, 700 covers most needs)
Token savings

A lean token system IS Occam's Razor applied. 8 spacing values, 5 radius values, 3 shadow levels. Resist adding tokens "just in case."

⏳
Heuristic

Parkinson's Law

"Work expands to fill the time available."

In UI: forms expand to fill available space. Constrain input sizes to suggest appropriate response length.

When to apply

Form fields, text areas, time limits, task interfaces

How to implement
  • Size inputs to expected content: short input for zip code, wide for email
  • Use max-width constraints and character counters
  • Break large tasks into small, time-bounded steps
Token savings

Input width tokens: --input-sm: 120px, --input-md: 280px, --input-lg: 100%. Consistent sizing without one-off widths.

Gestalt

Law of Prägnanz

"People interpret complex shapes as simple forms."

The brain prefers simple, regular shapes. Use clean geometric forms, consistent radii, and symmetry in your components.

When to apply

Icons, logos, card shapes, button styles, overall layout geometry

How to implement
  • Use a consistent border-radius scale (not random values)
  • Align elements to a grid for regularity
  • Keep icon styles uniform (all rounded or all sharp, not mixed)
Token savings

Your --radius-* scale IS Prägnanz. Five values, applied consistently, create the visual regularity the brain craves.

/* Pattern: Toggle with data attributes */
document.querySelectorAll('[data-toggle]')
  .forEach(trigger => {
    trigger.addEventListener('click', () => {
      const target = document.getElementById(
        trigger.dataset.toggle

Course Notes — Emil's Easing Rules

▼

The #1 rule: Almost never use CSS ease or ease-in. Default to ease-out for user-triggered interactions — the fast start gives a feeling of responsiveness.

/* Easing tokens — Emil's production toolkit */
:root {
  /* Primary: use this 80% of the time */
  --ease-out:       cubic-bezier(0.16, 1, 0.3, 1);
  --ease-out-quart: cubic-bezier(0.25, 1, 0.5, 1);

  /* Springs: for playful, physical interactions */
  --ease-spring:    cubic-bezier(0.34, 1.56, 0.64, 1);
  --ease-spring-sm: cubic-bezier(0.22, 1, 0.36, 1);

  /* Special: drawers, sheets, iOS-feel */
  --ease-ios:       cubic-bezier(0.32, 0.72, 0, 1);

  /* Symmetric: looping, auto-play, loading */
  --ease-in-out:    cubic-bezier(0.65, 0, 0.35, 1);
}

/* When to use which:
   User click/tap → ease-out (fast start, slow end)
   Drawer/sheet   → ios-drawer (heavier, weighted)
   Bounce/delight → spring-bounce (overshoots)
   Auto-play/loop → ease-in-out (symmetrical)
   Spinners       → linear (constant speed)
   NEVER          → ease-in for UI (feels sluggish) */

Course Notes — Duration Tokens

▼

Emil's golden rules for timing:

1. Nothing in UI should be longer than 500ms. Anything slower and users feel like they're waiting.

2. Exit animations should be faster than entrances — the user is done and wants to move on.

3. Scale duration with distance/size. A tiny tooltip can be 100ms. A full-screen modal needs 300ms.

/* Duration tokens — a complete scale */
:root {

  --duration-fast:   150ms;  /* buttons, toggles, tooltips */
  --duration-normal: 250ms;  /* dropdowns, accordions */
  --duration-medium: 350ms;  /* modals, panels */
  --duration-slow:   500ms;  /* drawers, page transitions */
}

/* Enter: 300ms, Exit: 200ms (faster!) */
.modal-enter { transition: all var(--duration-medium) var(--ease-out); }
.modal-exit  { transition: all var(--duration-fast) var(--ease-out); }

transform: translateX(20px);   /* move */

@keyframes fadeSlideIn {

Chapter 3

The Big Little Details

The micro-interactions and polish that separate good UI from great UI — staggered lists, animated presence, reduced motion, and knowing when not to animate.

💡 Visual — Emil's Practical Animation Tips

15+ tips distilled from the course. Click any tip to expand the implementation detail.

1️⃣
Always default to ease-out, not ease or linear
transition: transform 250ms cubic-bezier(0.16, 1, 0.3, 1) — The fast start gives users instant feedback. CSS's default ease has a slow start that feels sluggish.
2️⃣
Make exits faster than entrances
Enter at 250–350ms, exit at 150–200ms. The user has already committed to closing — don't make them wait. This is the Peak-End Rule in practice.
3️⃣
Only animate transform and opacity
These are GPU-composited and run at 60fps. Animating width, height, top, left, margin, or padding triggers layout recalculation — causing jank. Use transform: scale() instead of width.
4️⃣
Stagger list items with transition-delay
.item:nth-child(1) { transition-delay: 0ms; }
.item:nth-child(2) { transition-delay: 50ms; }
.item:nth-child(3) { transition-delay: 100ms; }
Or use CSS custom properties: style="--i: 2" → transition-delay: calc(var(--i) * 50ms). Keep total stagger under 300ms.
5️⃣
Use spring easing for bouncy, physical interactions
cubic-bezier(0.34, 1.56, 0.64, 1) — The values above 1.0 in the second/fourth params create overshoot. Great for toggles, notifications popping in, and reaction animations.
6️⃣
Respect prefers-reduced-motion — always
@media (prefers-reduced-motion: reduce) {
  * { animation-duration: 0.01ms !important;
      transition-duration: 0.01ms !important; }
}
Don't remove animations entirely — use subtle opacity fades as fallback.
7️⃣
Don't animate things that don't need it
Not everything needs to move. Static labels, disabled elements, and informational content should stay still. Animation should serve purpose: guide attention, provide feedback, show spatial relationships, or create delight at key moments.
8️⃣
Use transform-origin for SVG and icon animations
SVGs default to top-left origin. Set transform-origin: center center for rotation/scale, or use a specific point like transform-origin: 50% 80% for a "hinge" effect on icons.
